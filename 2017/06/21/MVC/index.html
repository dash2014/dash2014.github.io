MVC&MVP&MVVM

发表于 2017-06-21 | 分类于 tech

什么样的框架更适合什么样的程序，一万个人心中有一万个哈姆雷特，之前有一段时间看了一些篇文章，抽象出来的框架无外乎MVC，MVP以及MVVM，今天正好抽空写一下我对这几种框架的感觉，有错误的地方欢迎指出。

在框架设计中，需要考虑的事情通常都只有这样几件，如何平衡决定了框架设计者的能力：

  • 程序的复杂度，包括可理解程度、代码量以及可维护的程度
  • 程序的可复用程度，是否足够解耦，其颗粒化是否可以达到较好的复用
  • 模块间通信，通信是否足够简单，方便
  • 可扩展性，是否拥有足够的扩展空间，以最小化的力度接受后续产品及业务的变更

MVC

MVC在iOS开发中应用之广，其中Modal， View， Controller很好的将数据与视图进行分离，好吧，上苹果开发者一定见过的一张神图：

这张图中，苹果给我们规范了MVC的标志写法

  • Model通过Notification和KVO的方式与Controller交互，Controller直接访问Model
  • Controller与View通过delegate的方式交互，Controller为View提供数据源，View向Controller提供事件发生
  • Model与View不能相互通信

现实情况，远比理想中的模型复杂的多，这套MVC存在着很多必然的问题：

  • 过于庞大的Controller，其除了负责View和Model的交互，还要响应各种代理事件，完成各种业务逻辑，甚至包括很多Model的组装和拼接，View的各种生命周期等等
  • 过于轻量的Model，除了定义的几个属性，别无其他
  • 网络层置于何地，Model层嘛？但是Model只是纯粹的数据定义，Controller层？OMG~放过这个可怜的Controller吧，但事实是网络层大部分都写在了Controller中

另外，更现实的情况是，你会通常用Model去初始化View，例如cell的配置，因此MVC只是理论上的。

结合我之前总结的框架设计者需要考虑的几件事情，MVC模式的特征大概是这样的：

  • View和Model复杂度低，Controller复杂度极高
  • View和Model复用度高，Controller无法复用
  • 模块间通信理解简单，View与Controller交互，Model与Controller交互
  • 可扩展性，Controller的扩展性极低，里面业务量巨大

MVP

MVP，MV都没变化，P代表Presenter，可以理解为一个表示层吧，话不多说，上图：

这张图简要的介绍了MVP的基本结构，我大致描述一下：

  • View层只负责数据展示，和从前没什么变化
  • View和Controller之所以统一作为PassiveView，是因为Controller负责View的布局和管理
  • 抽象出一个Presenter层，其通过Controller以及Model来进行初始化，同时提供更新接口，更新Controller中的View的数据，以及更新Model数据
  • Model仍旧仅仅负责数据

OK，那MVP有什么问题存在呢？

  • 依旧是过于轻量级的Model
  • Presenter层用来协调View的状态更改和Model的状态更改，以及Model和View展示的数据适配，Presenter层的代码量会比较大
  • 网络层依然存放在Controller中

因此总结一下MVP的特征大概如下：

  • View和Model复杂度低，Controller复杂度降低，但代码量并没有减少太多，因为Presenter层更新仍旧需要调用Controller中的updateView的方法来更新View的状态
  • View和Model复用度高，Controller可以适度复用，Presenter层无法复用，但业务更改更集中在Presenter层，实现了业务和展示分离
  • 模块间通信理解略复杂，View与Controller交互，Controller与Presenter层交互，Presenter层与Model交互
  • 可扩展性，Controller的扩展性较高，Presenter层业务量增大，但扩展性尚可

PS: 看见很多人介绍了一个变种，说Presenter层可以协调View和Model做绑定，个人觉得这样会导致View和Model的可复用性极度降低，比如A中更改了Model的数据，之后又在B中更改了同一个Model的数据，维护起来太难了，不推荐。

MVVM

不得不吐槽一下MVVM的概念简直被玩坏了，随便增加个model做个绑定就是MVVM了。。。好吧，VM其实就是ViewModel的简写，话不多说，继续上图。

这张图简单的展示了MVVM的概念，看着和MVP很像是吧，那么简单的说一下MVVM吧

  • View层只负责数据展示，和从前没什么变化
  • View和Controller之所以统一作为View，同样是因为Controller负责View的布局和管理
  • 抽象出一个VM层，其通过Controller以及Model来进行初始化，其用来更新Model的数据，同时将VM与View进行绑定，当VM发生变更的时候自动变更View的状态，注意哦，不是绑定Model，而是VM~
  • Model仍旧仅仅负责数据

OK，那MVVM有什么问题存在呢？

  • 依旧是过于轻量级的Model
  • VM和Presenter层并没有什么特别大的改进，依然具有大量的代码
  • 网络层依然存放在Controller中

因此总结一下MVVM与的特征大概如下，其于MVP的特征大致相似：

  • View和Model复杂度低，Controller复杂度降低
  • View和Model复用度高，Controller可以适度复用，VM层无法复用，但业务更改更集中在VM层，实现了业务和展示分离
  • 模块间通信理解略复杂，View与Controller交互，Controller与VM层交互，VM层与Model交互
  • 可扩展性，Controller的扩展性较高，VM层业务量增大，但扩展性尚可
  • 实现了自动绑定，Controller可以不用操心View的数据变化

框架总结

  • 业务简单，MVC足矣
  • 业务略复杂，数据通讯及交互不多，可适度采用MVP
  • 业务复杂，数据通讯多，可考虑MVVM
  • 任何一种框架都只是能够有限的解决某一时刻的某种问题，不要为架构而架构
  • 任何一种框架在解决一种问题的时候，通常会带来另一种问题，这就好比锁，可以保证你的数据安全，但频繁加锁又会影响性能一个道理。

我们工程中的框架分层

上述只是简单的介绍了一下MVC、MVP以及MVVP的描述，并没有涉及到更多复杂的情况，比如网络层，缓存层，数据处理等

一直认为一个比较好的框架结构，原谅我并没有绘制图~

  • Controller负责初始化Manager层，同时布局各种View
  • View层仍旧只负责展示数据
  • Manager层负责从网络或者缓存中读取，以及生成AdaptModel，并将其提供给Controller层
  • AdaptModel层负责根据Manger的网络回调或者缓存读取到的BaseModel进行数据整理和拼接等，可以将其看作BaseModel的表示层
  • BaseModel就是数据持久化的Model

当然这样做的好处就是展示数据，原始数据，网络，视图分离的比较开，颗粒化比较好，当然这样也是存在很多问题的，但是在实际工程中应用起来确实还算适应，那么它的问题是什么？

  • Model层简化
  • 如果业务量很小，直接用Model就可以标识，AdaptModel无用，增加了很多额外的代码量

仍旧是那句老话，任何框架都有其局限性，具体还要根据实际情况进行选择~
